### Invitations: admin-minted signup codes

- Schema: `invitationsTable` in `schema.ts` has a required `code` (unique), a required `status: textEnum(INVITATION_STATUSES)` that drives the state machine, a nullable `usedAt: timestamp()`, and two FK columns `usedBy` / `createdBy` both typed `id("user").references(() => userTable.id)`. The FKs were originally plain `text()` and were retyped when the admin dashboard shipped — reads now come back narrowed to `Id<"user">`, and the `signUpWithInvitation` mutation writes `result.user.id as Id<"user">` rather than a username string. There's no explicit `createdAt` column; kitcn aliases it to Convex's `_creationTime` (a number), so the list projection surfaces it as a raw ms timestamp instead of a `Date` (contrast with `usedAt`, which is user-declared and comes back as `Date`).
- Status state machine (`shared/tables/invitations.ts`, `INVITATION_STATUSES = ["active", "used", "revoked"]`): newly minted rows start `active`; `signUpWithInvitation` atomically transitions `active → used` (and fills `usedAt` + `usedBy`); an admin revoke transitions `active → revoked` (soft-delete, preserves audit trail). `used` rows are terminal — revoke of a used row throws `BAD_REQUEST`. The row is **never hard-deleted**, so the `code` unique constraint stays in force even after revoke. If a future requirement needs to reuse revoked codes, the migration path is a partial unique index like `(code, status = 'active')` — inline comment in the shared module spells this out. Signup now discriminates on `status` (checking both `used` and `revoked` branches) rather than `usedAt != null`, because a revoked row has a null `usedAt`.
- Procedures (`functions/invitations.ts`) are all `authQuery` / `authMutation.requires(["admin"])`:
  - `list` — page-number pagination via `findMany({ orderBy: { createdAt: "desc" }, offset: page * pageSize, limit: pageSize + 1 })` (no `where` — admin sees every row). Capped at `INVITATION_LIST_PAGE_SIZE_MAX = 100`. Ordering uses Convex's system `_creationTime` so no secondary index is required. Returns `ok({ items, page, pageSize, hasMore })` — `hasMore` is the `+ 1` look-ahead trick (fetch one extra row, slice it off, flip the flag). After slicing, the procedure collects the union of `createdBy` and `usedBy` ids from the kept rows, batches them through a `findMany + inArray` lookup against `userTable` (chunk size 30 — Convex's ~64-level JSON recursion limit on kitcn's left-folded `or` compilation; chunk size 30 is the empirically-safe max), and emits `createdByName` / `usedByName` alongside the raw ids. Rows whose user was deleted resolve to `null` names and the UI falls through to a raw id / em-dash.
  - `count` — O(1) via `ctx.orm.query.invitations.count()`. Because there's no filter (admins see every row regardless of status), no `aggregateIndex` is needed — the plain count syscall suffices.
  - `create` — admin mints a code. If `input.code` is supplied, it's validated against `invitationCodeInputSchema` and written verbatim (throws `CONFLICT` on duplicate). If omitted, the backend generates a 12-character code from a 32-char alphabet (`ABCDEFGHJKLMNPQRSTUVWXYZ23456789` — deliberately excludes `0/O`, `1/I/L` to avoid ambiguity when a user types it) and retries up to 10 times on collision. `createdBy` is written as `ctx.user.id as Id<"user">`.
  - `revoke` — soft-revoke by setting `status = "revoked"`. Rejects with `BAD_REQUEST` on a non-`active` source status (differentiated Chinese messages for `used` vs `revoked`); `NOT_FOUND` if the row doesn't exist.
- Frontend (`apps/dashboard/src/routes/_authenticated/invitations/`): the route is a self-contained FSD-style slice — `-components/` (create + revoke dialogs, data-table + columns, row actions, status badge), `-lib/status-labels.ts`, `-model/` (row type, pagination reducer). Pagination state is `{ pageSize, pageIndex }` — changing `pageSize` resets `pageIndex` to 0 as a coupled invariant. Every page change is a single `crpc.invitations.list.queryOptions({ page, pageSize })` round-trip (server resolves the page in one offset-based scan), with `placeholderData: keepPreviousData` so the table renders prior rows during the fetch instead of flashing empty. After a successful create, the page dispatches `change_page: { pageIndex: 0 }` so the newly minted row — which lands at the top of the descending-order list — is visible without manual navigation.
