## Conventions worth knowing

- **When dealing with kitcn, read the bundled skill docs first.** kitcn ships authoritative skill docs at `packages/backend/node_modules/kitcn/skills/convex/SKILL.md` (with a `references/` tree alongside), versioned in lock-step with the installed package. Before claiming how kitcn behaves, advising a CLI invocation, or interpreting a builder API, consult those skills — assumption-based answers have already caused production mis-routing here. Full source-of-truth order in [backend-architecture.md](backend-architecture.md).
- **Package manager is bun 1.3.11** (pinned via `packageManager`). Use `bun install`, not npm/yarn/pnpm.
- **Oxc** is the formatter + linter — `oxfmt` for formatting, `oxlint` for lint. `bun run check:fix` runs `oxfmt && oxlint --fix`. Both tools share the same ignore list (`.oxfmtrc.json` and `.oxlintrc.json`): `packages/ui/src/{components,hooks,lib}`, `routeTree.gen.ts`, all `_generated`/`generated` dirs, and `shared/api.ts` — don't try to reformat those. Unlike the old Biome setup, oxfmt also formats `*.md` and `*.css` (tables get column-aligned, CSS gets tab-indented), so a `check:fix` pass will touch docs and styles — commit the churn with whatever change triggered it. `.oxlintrc.json` enables `typescript/no-explicit-any` as an error; silence intentional uses with `// oxlint-disable-next-line typescript/no-explicit-any -- <rationale>` (see `convex/lib/responses.ts` for the canonical CRPCError example).
- **Query loading states: use `isPending`, not `isLoading`.** TanStack Query's `isLoading` is `isPending && isFetching` — during SSR hydration the query hasn't started fetching yet, so `isLoading` is `false` while `isPending` is `true`. If the component's "no data" branch renders interactive UI (e.g. a form), users see a flash of that UI before the skeleton/spinner kicks in. Always gate loading states on `isPending` to cover the full window from first render to data arrival.
- **Prefer `useReducer` over multiple `useState` when state pieces are coupled.** A single `useState` per atom is fine for genuinely independent values. Reach for `useReducer` as soon as two or more pieces of state (a) share an invariant enforced across updates (e.g. "changing page size must reset page index", "editing a field clears its error"), (b) need atomic transitions the caller would otherwise have to remember to do in a specific order, or (c) are updated together from multiple code paths that would drift. Put the reducer in the slice's `-model/` folder alongside its state and action types — one file per form/concern (the dashboard's `invitations/-model/pagination.ts` is the canonical example: `{ pageIndex, pageSize }` with the invariant that changing `pageSize` resets `pageIndex` to 0). The action types are discriminated unions (`type: "set_field" | "set_errors" | ...`) so the reducer's `switch` is exhaustive; idempotency guards inside the reducer (`if (state.x === action.x) return state`) are the canonical way to survive StrictMode double-invokes. The reducer file owns any derived types that describe the state shape — consumers import the type from the model file directly, not through the component.
- **One source of truth, no pass-through re-exports.** If a Zod schema, enum, or type already lives in `@repo/backend/shared/...` (or any other canonical home), import it **directly** at the consumer site. Don't create an intermediate module whose entire purpose is `import { x } from "canonical"; export { x };` — re-derived types (`z.infer<typeof x>`) should be declared at the single site that needs them, not paraded through a chain of modules. Duplicating a schema with slightly different error messages or `.trim()` is the same sin in a different disguise — push the Chinese messages / normalizations into the canonical schema (see how `usernameSchema` in `shared/tables/user.ts` carries its Chinese error messages) and let both the backend procedure and the frontend form consume the one schema.
- **Every procedure `.input()` binds to a shared schema — never an inline `z.object({...})`.** Each kitcn `.input(...)` call must pass a named schema exported from `packages/backend/convex/shared/tables/<table-name>.ts` (or another `shared/...` module). Follow the `<procedureName>InputSchema` naming: `signUpWithInvitationInputSchema`, `bootstrapAdminInputSchema`, `createInvitationInputSchema`. Why: the frontend imports `@repo/backend/shared/...` for form validation and types — an inline backend schema forces either a duplicate validator on the frontend (which drifts) or a types-only round-trip through the generated API surface (which gives you `z.infer` but not the validator). Put primitive constraints (e.g. `USERNAME_MIN_LENGTH`, `PASSWORD_MIN_LENGTH`) alongside the schema that uses them so HTML form attributes and the Zod validator pull the same numbers. Compose across tables by importing a sibling schema — `signUpWithInvitationInputSchema` in `shared/tables/user.ts` imports `invitationCodeInputSchema` from `./invitations`.
- **Feature branches, not main.** Local work happens on a branch; push to `main` triggers Cloudflare + Convex deploys, so don't push speculative work there.
- After any schema change in `packages/backend/convex/functions/schema.ts`, run `bun run codegen` and commit the regenerated files together with your source change. Convex will also demand a migration on next dev/deploy if you drop or rename a table.
- **kitcn default-target asymmetry**: `bunx kitcn deploy` defaults to **prod**, but `bunx kitcn migrate <subcmd>` and `bunx kitcn aggregate <subcmd>` both default to **dev** unless you pass `--prod` (Convex Cloud) or `--env-file .env.prod` (self-hosted) — this holds **even when `CONVEX_DEPLOY_KEY=prod:...` is exported**. Mis-targeting here is how you end up with a prod schema change and no backfill data, or a rebuilt aggregate in dev when you wanted prod. Full matrix in [`docs/kitcn-cli-guide.md`](kitcn-cli-guide.md); migration workflow in [`docs/MIGRATION.md`](MIGRATION.md).
