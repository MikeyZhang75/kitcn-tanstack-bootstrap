### Frontend: TanStack Start + TanStack Router

- Entry: `apps/web/src/router.tsx` creates the router (`defaultPreload: "intent"`). Route tree is file-based and generated into `routeTree.gen.ts` (gitignored).
- Route segments `_authenticated` and `_public` are **layouts**, not URL prefixes:
  - `_authenticated.tsx` runs the session gate in `beforeLoad` via `fetchSessionClaims()` (see [auth](auth.md)) and renders the **shared authenticated shell** (`SidebarProvider` + `AppSidebar` + `SidebarInset` with a header containing `SidebarTrigger` + vertical separator). All pages under `_authenticated/` render inside this shell.
  - **Shell scroll containment**: the shell caps itself at viewport height so long pages scroll inside the content region, not the page. `SidebarProvider` is `h-svh overflow-hidden` (caps height + clips `SidebarInset`'s `m-2` inset margin from `variant="inset"`, which would otherwise push the layout past the viewport), `SidebarInset` is `min-h-0` (lets the flex child shrink below its content), and the `<Outlet />` container is `min-h-0 flex-1 overflow-auto`. If you reintroduce `min-h-screen` / `min-h-svh` on a page wrapper or drop any of these classes, you'll get a double scrollbar back — the whole page scrolls instead of just the content area.
  - `_public.tsx` is a bare `<Outlet />` — no guard. Pages that need "bounce already-signed-in visitors" (like `/auth`) put that check in their own `beforeLoad`; pages that should be reachable regardless of auth state (like `/access-denied`) just live here and inherit nothing.
- Current routes under `_authenticated` **in `apps/web`**: `index.tsx` (dashboard placeholder). The sidebar in `components/app-sidebar.tsx` has one nav item: 仪表盘 (`/`). Add new feature slices as sibling folders under `_authenticated/` and extend the `navItems` array.
- **`apps/dashboard` is an admin-only app sharing plumbing with `apps/web`.** Present routes: `_authenticated/index.tsx` (dashboard placeholder) and `_authenticated/invitations/index.tsx` (invitation-code management — offset-paginated list, create dialog, revoke dialog). `app-sidebar.tsx` has two nav items: 仪表盘 (`/`) and 邀请码 (`/invitations`); the sidebar header title is "管理后台". Shared **app-level** plumbing (`components/providers.tsx`, `routes/__root.tsx`, `routes/_public/{auth,access-denied}.tsx`, `routes/api/auth/$.ts`, `styles.css`, `tsconfig.json`) is byte-identical across the two apps. When you change any of those shared files, **mirror the edit into both apps in the same commit** — there's no shared-frontend package for app-level code today and the duplication is intentional (keeps each app's bundle scoped to its own audience's UI). Pure shadcn primitives, the `cn` helper, `use-mobile`, the globals stylesheet, and three first-party wrappers (`DataTable`, `LoadingButton`, `NavUser`) were extracted into `packages/ui` — see [UI components](ui-components.md). The Convex/Better-Auth plumbing (cRPC, query client, error normalizer, JWT claims, auth-server handler, `fetchSessionClaims`/`readSessionClaims`) was extracted into `packages/app-convex` — only `lib/convex/auth-client.ts` remains per-app because the two apps export different Better Auth mutation hooks. `app-sidebar.tsx` is per-app (different nav items + sidebar header title) and owns the small `AppNavUser` helper that wires `authClient.useSession()` into the shared `<NavUser>` UI. The Vite config lives in `packages/vite-preset` as `createAppConfig()` so each app's `vite.config.ts` is a 3-line delegation, and plugin / `optimizeDeps` changes only happen in one place.
- **Route-local code lives in sibling `-`-prefixed folders (FSD-style slices).** TanStack Router's file-based router skips any path segment starting with `-`, so folders like `-components/`, `-lib/`, and `-model/` are invisible to the route generator and a natural home for code that only one route uses. **Each route directory is a self-contained slice** with its own `-components/`, `-lib/`, `-model/` sub-folders — slice boundaries are per-route, not shared across siblings. The `apps/dashboard/src/routes/_authenticated/invitations/` slice is the canonical reference: `index.tsx` (route entry) with `invitations/-components/` (dialogs, columns, data-table, row actions, status badge), `invitations/-lib/` (status labels), and `invitations/-model/` (row type, pagination reducer).

  Prefer this per-route isolation over dropping single-use code into `apps/<name>/src/components/` or `apps/<name>/src/utils/`. Code shared across multiple routes (like `DataTable`) belongs in `@repo/ui/components/custom-ui/`.

- Providers tree (`components/providers.tsx`): `AppConvexProvider` (from `@repo/app-convex/convex-provider`) wraps `ConvexAuthProvider` → `QueryClientProvider` → `CRPCProvider`. The app's `providers.tsx` passes its per-app `authClient` in as a prop; the package owns the `ConvexReactClient` instance, the query-client singleton, and the cRPC provider. It also mounts Sonner's `<Toaster position="top-center" richColors />`. Toasts are the standard channel for surfacing mutation errors (don't render inline error text).
- **Stale-chunk self-heal in `__root.tsx`**: a module-level `vite:preloadError` listener triggers a single `window.location.reload()` (gated by a `sessionStorage` flag so it can't loop) when an old `index.js` references a hashed route chunk that no longer exists post-deploy. It looks like incidental event wiring — **don't strip it during cleanup**. It's the only thing protecting users on a stale tab from a Cloudflare Pages 404 after a redeploy. The flag self-clears on a successful `load`, so a future genuine stale chunk can still trigger recovery.
- Env vars are validated through `@t3-oss/env-core` in `apps/web/src/env.ts`. Required at build and runtime: `VITE_CONVEX_URL`, `VITE_CONVEX_SITE_URL`, `VITE_SITE_URL` (these are also declared as Turbo build inputs in `turbo.json`).
